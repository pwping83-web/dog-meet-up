# 06 — 특허·지식재산(IP) 가능 여부, 출원 개요

> **법률·변리 자문이 아닙니다.** 출원·권리화 전에는 반드시 **변리사·특허법률사** 상담을 받으세요. 아래는 사업계획서·내부 검토용 **일반 정리**입니다.

---

## 6.1 왜 문서로 남기는가

- 지원사업·투자 질문에 **“IP 전략이 있나요?”**에 답할 수 있게 합니다.  
- 개발자가 나중에 **오픈소스·외주·직원**과 귀속을 정리할 때 기준점이 됩니다.

---

## 6.2 특허(발명) — 소프트웨어·플랫폼에서 흔한 현실

| 항목 | 내용 |
|------|------|
| **심사 경향(일반론)** | 한국·주요국에서 **“컴퓨터 프로그램 자체”**만 나열한 출원은 **거절 사유**가 되기 쉽습니다. **기술적 과제**를 **구체적 수단**으로 해결했다는 서술이 중요합니다. |
| **상대적으로 검토할 만한 축** | 단순 “DB에 필드 추가”가 아니라, **측정 가능한 효과**(지연 감소, 오조작 방지, 부하 분산 등)와 **비자명성**이 뒷받침될 때 논의 가치가 있습니다. |
| **플랫폼·운영 정책만** | “인증 받은 사람만 노출” 같은 **비즈니스 규칙·UI**만으로는 **특허성 판단이 불리한 경우**가 많습니다. 같은 아이디어를 **다른 기술로 구현**할 수 있기 때문입니다. |

**댕댕마켓 코드베이스를 염두에 둔 질문 예시(출원 전 변리사와 논의)**

- Edge에서 JWT 검증 방식 + 특정 RPC 조합이 **보안·오남용 방지**에서 **구체적으로 무엇을 개선**하는가?  
- RLS·노출 기한·결제 웹훅이 **하나의 기술적 흐름**으로 묶일 때, 선행기술 대비 **진보성** 주장이 가능한가?

→ **결론:** “우리 앱 전체를 특허낸다”기보다, **특정 알고리즘·동기화·검증 파이프라인**처럼 **좁고 깊게** 쪼개서 조사하는 편이 일반적입니다.

---

## 6.3 특허 말고 같이 보는 IP (실무에서 더 자주 씀)

| 유형 | 용도 | 댕댕마켓 관련 예 |
|------|------|------------------|
| **상표** | 서비스명·로고·슬로건 | “댕댕마켓”, 캐릭터·워드마크 — **출원 전 유사 상표 검색** 권장 |
| **저작권** | 소스·UI·문서·이미지 | Git 커밋·계약서에 **귀속(회사/개인)** 명시 |
| **실용신안** | 형·구조 중심(디바이스·기구에 가까움) | 순수 웹서비스만이면 발명보다 **득이 적은 경우**가 많음 |
| **영업비밀** | 공개하지 않을 운영 노하우·프롬프트·가격 정책 | NDA·접근 통제 |

## 6.4 출원·등록까지의 흐름(개략)

1. **선행기술 조사** — 키워드·IPC 코드로 유사 특허·논문 검색(변리사 의뢰가 정확).  
2. **청구범위 초안** — “무엇을 독점하고 싶은지”를 문장으로 쪼갬.  
3. **명세서 작성** — 발명의 상세한 설명, 도면, 실시예.  
4. **출원** — 특허청 전자출원 등.  
5. **심사 청구** — 출원 후 일정 기간 내 심사 청구(미청구 시 취하 등 — 법령 확인).  
6. **거절이유 통지 → 의견서·보정** — 반복 가능.  
7. **등록 결정 → 등록료 납부 → 특허권 발생.**

**비용·기간**은 출원국·청구항 수·분쟁 여부에 따라 크게 달라집니다. 견적은 변리사 사무소에 요청하세요.

---

## 6.5 지원사업·정부 과제와의 연계

- 일부 과제는 **IP 창출 계획·출원 실적**을 평가에 반영합니다.  
- **IP 나래**, **TIPS** 등 프로그램은 자격·일정이 수시로 바뀌므로 **당해 연도 공고문**을 기준으로 하세요.

# 댕댕마켓(Dog Meet-up) — 지원사업·사업계획 첨부용 프로젝트 요약

> **문서 목적:** 정부·지자체·액셀러레이터 등 지원사업 제출 시 서비스 이해를 돕기 위한 기술·사업 요약입니다.  
> **법률·특허 판단:** 본 문서의 「특허」 관련 서술은 일반 정보이며, 법률 자문이 아닙니다.

---

## 1. 서비스 한 줄 소개

**반려견 커뮤니티·돌봄 연결 웹앱**으로, 모임 글(모이자·만나자), 돌봄 맡기기, **운영자 인증 기반 인증 보호맘** 프로필·예약, Stripe 결제·노출 정책을 **Supabase(PostgreSQL + Auth + RLS + Edge Functions)** 위에 구현한 **PWA(Vite + React)** 서비스입니다.

---

## 2. 사회적 필요성 및 문제 인식

| 영역 | 내용 |
|------|------|
| **산책·모임** | 견주 간 지역 기반 만남·소모임 수요는 크나, 신뢰·안전·일정 조율을 한곳에서 다루기 어렵습니다. |
| **돌봄** | 맡기기·맡아주기 정보가 분산되면 검증된 돌봄 제공자를 찾기 어렵습니다. |
| **운영** | 커뮤니티 앱은 **스팸·비인증 돌봄** 리스크가 있어, **운영 주체의 인증·노출 정책**이 있을 때 신뢰도가 올라갑니다. |

본 프로젝트는 **글·프로필·결제·RLS(행 단위 보안)** 를 묶어 **운영 가능한 최소 단위의 플랫폼**을 목표로 합니다.

---

## 3. 사업·서비스 목표

1. **지역 기반 반려견 모임·정보 교류** 활성화(모이자·만나자·돌봄 카테고리).  
2. **인증 보호맘** 등 돌봄 제공 측의 **등록 → 운영자 인증 → 목록 노출(프로모/유료 RLS)** 으로 **신뢰 가능한 돌봄 탐색** 제공.  
3. **Stripe** 기반 **유료 노출·돌봄 예약 결제** 등 수익·운영 구조 확보.  
4. **모바일 우선 UI**·PWA로 접근성 강화.

## 4. 주요 기능(사용자 관점)

| 구분 | 설명 |
|------|------|
| **탐색·피드** | 홈·탐색에서 모임·돌봄 관련 콘텐츠 탐색, 검색. |
| **모이자·만나자** | 카테고리별 모임 글 작성·상세·참여 신청 흐름. |
| **인증 돌봄** | 댕집사(목업/향후 확장), **인증 보호맘**(DB `certified_guard_moms`), 맡기는 사람(돌봄 글) 탭 구조. |
| **보호맘** | 등록(`/guard-mom/register`), 상세·예약·결제 플로우, **관리자 인증** 후 공개 노출(RLS·프로모 플래그 연동). |

| **관리자** | 지정 관리자만 접근, 돌봄·결제·예약 조회, **RPC로 인증 시각(`certified_at`) 반영**. |
| **고객지원·알림** | 고객센터, 알림·채팅 UI(도메인별 구현 수준은 정책에 맞춰 기술). |

## 5. 작동 원리(시스템·데이터)

### 5.1 전체 구조

```
[사용자 브라우저 — Vite/React PWA]
        │
        ├─ Supabase Auth (JWT)
        ├─ PostgREST: 테이블 CRUD (RLS 적용)
        ├─ RPC: 예) admin_set_guard_mom_certified (SECURITY DEFINER)
        └─ Edge Functions: Stripe Checkout 생성, Webhook 처리
        │
[Supabase — PostgreSQL]
  · RLS로 행별 읽기/쓰기 제한 (본인·인증된 공개·관리자 정책 등)
  · 트리거로 일반 사용자의 certified_at 등 민감 필드 변경 제한, 관리자 예외
```

### 5.2 인증 보호맘 핵심 흐름

1. 사용자가 **보호맘 프로필**을 `certified_guard_moms`에 저장(본인 `user_id`).  
2. **운영자**가 관리자 화면에서 `admin_set_guard_mom_certified` RPC 호출 → `certified_at` 갱신.  
3. **SELECT 정책**(한시 무료: `certified_at`만으로 공개 / 유료: `listing_visible_until` 등)에 따라 **타 사용자 목록 조회** 가능 여부 결정.  
4. **Stripe**로 노출·예약 결제 시 Webhook으로 DB 권한·기한 필드 갱신(마이그레이션·함수와 연동).

### 5.3 기술 스택 요약

| 계층 | 기술 |
|------|------|
| 프론트엔드 | React 18, React Router 7, Vite, TypeScript, Tailwind CSS, PWA(vite-plugin-pwa) |
| BaaS·DB | Supabase(PostgreSQL, Row Level Security, Auth, Edge Functions) |
| 결제 | Stripe(Checkout, Webhook) |
| 기타 | EmailJS 등 알림·문의 보조 |

## 6. 차별화·강점(제출 시 활용 포인트)

- **운영 인증 + DB 정책(RLS) + 결제**를 같은 스키마로 묶어 **「인증된 돌봄만 노출」** 정책을 코드·DB로 일관되게 구현.  
- **관리자 전용 RPC·정책**으로 일반 클라이언트 `UPDATE` 우회·오조작을 줄임.  
- **프로모 무료 / 유료 노출** 전환을 마이그레이션·환경변수로 분리해 **사업 단계에 맞춘 운영** 가능.  
- **모바일 우선** UI·지역·거리 보조로 실사용 시나리오에 맞춤.

## 8. 지적재산권·「특허 사업」 가능 여부에 대한 일반 안내

### 8.1 특허(발명)

- 대한민국에서 **프로그램 자체(소스 코드 문장)** 는 특허 대상이 아니며, **컴퓨터 관련 발명**은 **기술적 과제를 기술적 수단으로 해결하는 것**으로서 **신규성·진보성** 등 요건을 충족할 때 보호될 수 있습니다.  
- **일반적인 웹 마켓플레이스·CRUD·JWT·RLS 조합만**으로는 진보성 인정이 어려운 경우가 많습니다.  
- **특정 기술적 효과**(예: 검증 가능한 분산·암호 결합, 측정·제어와 불가분하게 결합된 알고리즘 등)가 **명확한 과제-수단-효과**로 정리될 때 변리사와 상담 가치가 있습니다.

**→ 본 프로젝트가 “특허 가능/불가능”을 여기서 단정할 수 없습니다.** 제출용 문서에는 **「특허 출원 검토 예정(변리사 자문)」** 정도로 적고, **실제 출원·심사는 특허청·변리사**에 맡기는 것이 안전합니다.

### 8.2 특허 외 지적재산(현실적으로 활용하기 쉬운 항목)

| 유형 | 내용 |
|------|------|
| **상표** | 서비스명·로고(댕댕마켓 등) 브랜드 보호. |
| **저작권** | 소스 코드·문서·디자인 에셋(고용·위탁 계약 시 귀속 조항 정리). |
| **디자인** | GUI 일부는 의장 출원 검토 가능(요건은 별도). |

### 8.3 지원사업 서류에 쓸 때 문구 예시

> 본 과제는 소프트웨어 구현 중심이며, 핵심 로직·브랜드에 대해 **저작권·상표**를 우선 정리하고, 기술적 발명 요소가 확정될 경우 **특허 출원 여부를 변리사와 검토**할 예정입니다.
